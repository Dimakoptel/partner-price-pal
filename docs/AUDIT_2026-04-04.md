## 🔴 Критические проблемы

### 1. Партнёр попадает в рабочий интерфейс
**Проблема:** При регистрации нового партнёра он получает роль `user` по умолчанию (триггер `handle_new_user_role`). Пока админ не назначит роль `partner`, пользователь видит основной интерфейс компании вместо партнёрского кабинета.
**Решение:** Добавить выбор роли при регистрации. Пользователь с заявленной ролью «партнёр» сразу должен перенаправляться на экран ожидания партнёрского типа.

### 2. Нет назначения роли партнёра из админки
**Проблема:** В `UsersTab` нет UI для назначения роли `partner`, привязки `user_id` к карточке клиента и настройки скидок по категориям.
**Решение:** Расширить карточку пользователя в админке: добавить назначение роли, привязку к клиенту, управление скидками.

### 3. Нет удаления пользователей
**Проблема:** Администратор не может удалить пользователя из системы.
**Решение:** Добавить кнопку удаления с диалогом подтверждения. Использовать `supabase.auth.admin.deleteUser()` через edge function.

### 4. Партнёр не может использовать калькуляторы — но менеджер тоже не может из запроса
**Проблема:** Партнёр создаёт запрос с параметрами, но менеджер не может нажать «Просчитать» прямо из запроса. Нет интеграции запроса → калькулятор.
**Решение:** Добавить в интерфейс менеджера кнопку «Рассчитать» при просмотре запроса партнёра, которая вызывает логику калькулятора с переданными параметрами.

### 5. Юридические лица компании (реквизиты)
**Проблема:** Нет возможности задать реквизиты своих юрлиц (до 3) с логотипами для печатных форм.
**Решение:** Создать таблицу `company_entities` (name, inn, kpp, ogrn, legal_address, bank_*, logo_url, is_default). Использовать в печатных формах.

## 🟡 Важные недоработки

### 6. Регистрация не спрашивает роль
При регистрации нет поля «Кем вы являетесь» (сотрудник / дилер / агент / дизайнер). Администратор не видит, кем хочет быть новый пользователь.

### 7. Административный дашборд
Главная страница администратора идентична странице менеджера. Нет:
- Списка новых регистраций (ожидают одобрения) с быстрым действием
- Общей статистики по системе (заказы, производство, склад)
- Уведомлений о критических событиях
- Запросов партнёров, ожидающих ответа

### 8. Менеджерская часть для работы с запросами партнёров
В CRM нет вкладки «Запросы партнёров». Менеджер не видит запросы, не может отвечать, считать и конвертировать в заказ.

### 9. Удаление расчётов администратором
RLS-политика для DELETE на `saved_calculations` уже существует для админов, но в UI (HistoryPage) нет кнопки массового удаления для администратора.

### 10. Статусы заказа партнёра
Партнёр видит статус запроса, но после конвертации в заказ не видит статус производства и отгрузки своего заказа.

## 🟢 Рекомендации к улучшению

### 11. Безопасность
- RLS на `production_orders` использует модуль `calculator` вместо `production` — исправить
- Политики на `deal_activities` и `deals` используют роль `public` вместо `authenticated`
- Таблица `audit_log` не позволяет UPDATE/DELETE — **это правильно** (append-only)
- Storage bucket `partner-attachments` публичный — рассмотреть приватный доступ

### 12. Типизация
- `partner` передаётся как `as any` в `has_role` RPC-вызове — тип уже обновлён в enum, но код не обновлён

### 13. Печатные формы
- Нет выбора юрлица при печати документов
- Нет шаблонов КП/счёт/акт с подстановкой реквизитов

### 14. Мобильная версия
- AppLayout использует мобильное меню, но не все страницы адаптированы для мобильных устройств
- Партнёрский кабинет не тестировался на мобильных

### 15. Email-уведомления
- Нет уведомлений при регистрации нового пользователя
- Нет уведомлений партнёру при изменении статуса запроса
